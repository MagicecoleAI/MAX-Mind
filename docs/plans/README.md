## 목적

**개별 타스크(T-NNN)의 수행계획과 완료 보고를 한 파일에 연속 기록**한다.
착수 시점의 가설이 구현 과정에서 어떻게 검증/수정되었는지, 최종 결과가 무엇인지 한 번에 볼 수 있게 한다.

**이 디렉토리가 해결하는 문제**:
- PR 본문만으로는 "왜 그렇게 접근했나"가 남지 않음
- 같은 실수를 3개월 후 반복
- 완료된 타스크의 측정 결과가 흩어짐

---

## 파일 포맷

- 경로: `docs/plans/T-NNN-slug.md`
- 예: `docs/plans/T-001-docs-restructure.md`
- 슬러그는 영문 kebab-case 권장 (경로 가독성)

### GitHub Issues 병행 시 (Phase 3)
- `T-NNN`의 `NNN`을 **Issue 번호와 통일** (예: Issue #42 = T-042)
- 브랜치: `feat/T-042-slug`
- 커밋 메시지: `T-042: 메시지`
- PR 본문에 `Closes #42` 및 `See docs/plans/T-042-slug.md`

## 상태 생애주기

```
📋 계획 → 🔄 진행 → (🚧 블로커) → ✅ 완료
                              ↘ 🔴 중단/폐기
```

| 상태 | 조건 |
|------|------|
| 📋 계획 | 섹션 1~3 작성 완료, 착수 전 |
| 🔄 진행 | 착수, 진행 로그 추가 중 |
| 🚧 블로커 | 외부 의존으로 중단. orders에도 기입 |
| ✅ 완료 | 검증 기준 모두 충족, 섹션 5 기입 |
| 🔴 폐기 | 착수 전/후 불문 중단 결정. 사유 명시 |

---

## 작성 규칙

1. **1 타스크 = 1 파일**. 쪼개지 말 것 (대신 `- **후속 타스크**:`로 연결)
2. **배경(Why)과 검증 기준(DoD)은 착수 전에 반드시 작성**. 진행 중에 이게 없으면 탈선한다
3. **진행 로그는 날짜순 추가만**. 과거 수정 금지 (틀린 판단도 가치 있는 기록)
4. **최종 결과는 정량 수치로**. "개선됨" ❌ → "16ms → 8ms" ✅
5. **회고는 짧게 2~3줄**. 형식주의 경계
